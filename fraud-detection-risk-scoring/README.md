# Fraud Detection & Risk Scoring

## Описание проекта

Проект посвящён выявлению мошеннических транзакций и построению риск-скора. Основная сложность задачи — сильный дисбаланс классов: мошеннических операций значительно меньше, чем обычных.

Проект показывает не только построение ML-модели, но и аналитический подход к выбору порога, интерпретации ошибок и использованию результата в бизнес-процессе проверки транзакций.

---

## Бизнес-задача

Финансовой компании нужно заранее находить подозрительные операции, чтобы снижать потери и при этом не блокировать слишком много нормальных транзакций.

Практический сценарий:

    Новая транзакция → модель рассчитывает вероятность fraud → риск-скор попадает в очередь проверки → аналитик или антифрод-система принимает решение

---

## Данные

Используется датасет **Credit Card Fraud Detection**.

Особенности данных:

- транзакции европейских держателей карт
- признаки `V1`–`V28` получены после PCA-преобразования
- `Amount` — сумма операции
- `Time` — время относительно первой операции
- `Class` — целевая переменная: `1` для fraud, `0` для normal

---

## Постановка ML-задачи

Задача формулируется как бинарная классификация.

Целевая переменная:

```text
Class = 1, если транзакция мошенническая
Class = 0, если транзакция обычная
```

Для бизнеса важна не только accuracy, а баланс между:

- **Recall** — сколько fraud-транзакций удалось найти
- **Precision** — сколько найденных подозрительных операций действительно fraud
- **PR-AUC** — качество модели при сильном дисбалансе классов
- **Threshold** — управляемый порог для разных сценариев риска

---

## Этапы проекта

### 1. EDA

Проводится первичный анализ данных:

- размер датасета
- доля fraud-операций
- распределение сумм транзакций
- анализ признаков `Time` и `Amount`
- сравнение fraud и normal транзакций

### 2. Подготовка данных

На этом этапе выполняются:

- проверка пропусков
- масштабирование признаков
- train-test split со стратификацией
- подготовка baseline

### 3. Обучение моделей

Сравниваются несколько подходов:

1. Baseline-модель
2. Logistic Regression
3. Random Forest / Gradient Boosting

### 4. Оценка качества

Для оценки используются метрики, которые подходят для несбалансированных данных:

| Метрика | Зачем нужна |
|---|---|
| ROC-AUC | общее качество ранжирования |
| PR-AUC | качество на редком классе |
| Precision | точность fraud-срабатываний |
| Recall | полнота нахождения fraud |
| F1 | баланс precision и recall |

### 5. Risk Scoring

Вероятность fraud переводится в риск-скор. Это позволяет разделить операции на уровни риска:

| Уровень | Интерпретация | Действие |
|---|---|---|
| Low | низкий риск | пропустить автоматически |
| Medium | средний риск | отправить на мягкую проверку |
| High | высокий риск | отправить на ручную проверку |

---

## Ключевые выводы

- accuracy не подходит как основная метрика для fraud-задачи
- PR-AUC и recall важнее для оценки качества на редком классе
- выбор threshold зависит от стоимости ложных срабатываний и пропущенного fraud
- риск-скор удобнее для бизнеса, чем бинарный прогноз

---

## Бизнес-ценность

Проект показывает, как модель можно встроить в процесс антифрод-мониторинга:

- приоритизация транзакций для проверки
- снижение количества пропущенных fraud-операций
- настройка уровня риска под бизнес-цель
- прозрачная интерпретация ошибок модели

---

## Ограничения проекта

- признаки `V1`–`V28` обезличены, поэтому бизнес-интерпретация ограничена
- датасет исторический и не отражает все современные fraud-сценарии
- отсутствуют данные о пользователях, устройствах и merchant-категориях
- для production нужна онлайн-валидация и мониторинг drift

---

## Итог

Проект демонстрирует полный цикл решения fraud-задачи: анализ несбалансированных данных, обучение модели, оценка качества, подбор порога и перевод ML-результата в понятный риск-скор для бизнеса.

## Полный исполняемый код

В проект добавлен файл:

```bash
python src/full_analysis.py
```

Он выполняет весь цикл анализа:

- ищет CSV-файлы Kaggle в `data/raw`;
- если данных нет, создаёт демонстрационный датасет, чтобы код запускался сразу;
- проводит проверку качества данных;
- строит EDA и бизнес-метрики;
- сохраняет графики в `reports/figures`;
- сохраняет итоговые выводы в `reports/summary.md`.

Ноутбук в папке `notebooks/` содержит тот же полный пайплайн и подходит для просмотра на GitHub.
