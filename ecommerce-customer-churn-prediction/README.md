# E-Commerce Customer Churn Prediction

## Описание проекта

Проект посвящён анализу поведения клиентов e-commerce сервиса и прогнозированию оттока пользователей.

Цель проекта — построить модель, которая определяет клиентов с высоким риском ухода, а также выявить факторы, влияющие на churn. Такой подход применим в retention-аналитике, CRM-коммуникациях, персональных предложениях и повышении LTV.

Проект ориентирован на задачи продуктовой аналитики и аналитики данных: очистка данных, валидация, EDA, feature engineering, построение baseline, обучение моделей и интерпретация результатов.

---

## Бизнес-задача

В e-commerce часть клиентов перестаёт покупать после негативного опыта, снижения активности или потери интереса к продукту. Если заранее предсказать риск оттока, продукт может:

  * запускать персональные retention-кампании
  * давать скидки клиентам с высоким churn risk
  * повышать повторные покупки
  * улучшать клиентский опыт
  * снижать потерю выручки
  * развивать программу лояльности

Практический сценарий:

    Клиент давно не совершал покупку → модель оценивает churn risk → CRM-система отправляет персональное предложение

## Постановка ML-задачи

Задача формулируется как бинарная классификация на уровне клиента.

Целевая переменная:

    Churn = 1, если клиент ушёл в отток
    Churn = 0, если клиент остался активным

Модель должна вернуть вероятность оттока для каждого клиента. Далее эту вероятность можно использовать для приоритизации retention-кампаний.

## Этапы проекта

Проведена проверка качества данных:

  * анализ типов данных

### 2. EDA

Проведён исследовательский анализ данных:

  * распределение целевой переменной
  * churn rate по уровню удовлетворённости
  * churn rate по жалобам

Ключевые наблюдения:

  * короткий tenure и большой интервал с последнего заказа усиливают риск ухода

## Модели

В проекте сравниваются три подхода:

  1. Baseline rule-based model
  2. Logistic Regression
  3. Random Forest

### Baseline

Правило:

    если есть жалоба + низкий tenure
    или низкая удовлетворённость
    или давно не было заказа
    → predict Churn = 1

Результат:

| Metric | Value |
|---|---:|
| Precision | 0.198 |
| Recall | 0.496 |
| F1 | 0.283 |

Baseline полезен как простая бизнес-логика и точка отсчёта для ML-моделей.

---

### Logistic Regression

| Metric | Value |
|---|---:|
| ROC-AUC | 0.578 |
| Precision | 0.193 |
| Recall | 0.504 |
| F1 | 0.28 |

Логистическая регрессия хорошо подходит как интерпретируемая модель: можно анализировать направление влияния признаков на вероятность оттока.

---

### Random Forest

| Metric | Value |
|---|---:|
| ROC-AUC | 0.58 |
| Precision | 0.236 |
| Recall | 0.229 |
| F1 | 0.232 |

Random Forest показывает лучший баланс между качеством и устойчивостью к нелинейным зависимостям.

---

## Сравнение моделей

| Model | ROC-AUC | Accuracy | Precision | Recall | F1 |
|---|---:|---:|---:|---:|---:|
| Baseline | - | 0.578 | 0.198 | 0.496 | 0.283 |
| Logistic Regression | 0.578 | 0.565 | 0.193 | 0.504 | 0.28 |
| Random Forest | 0.58 | 0.746 | 0.236 | 0.229 | 0.232 |

Вывод:

  * baseline полезен как точка отсчёта
  * Logistic Regression даёт интерпретируемый результат
  * Random Forest лучше учитывает нелинейные зависимости
  * для retention-задачи особенно важны Recall и ROC-AUC, потому что бизнесу нужно находить клиентов с высоким риском оттока

## Feature Importance

Наиболее важные признаки для Random Forest:

  * `Tenure`
  * `CashbackAmount`
  * `WarehouseToHome`
  * `OrderAmountHikeFromlastYear`
  * `DaySinceLastOrder`
  * `NumberOfAddress`
  * `OrderCount`
  * `Complain`

Интерпретация:

  * жалобы клиента повышают вероятность оттока
  * низкая удовлетворённость усиливает churn risk
  * чем меньше tenure, тем выше риск ухода
  * большой интервал с последнего заказа может сигнализировать о потере интереса
  * cashback и история заказов помогают понять ценность клиента

## Product Recommendations

На основе результатов проекта можно предложить следующие продуктовые решения:

  * запускать персональные предложения для клиентов с высоким churn risk
  * отдельно работать с клиентами, у которых были жалобы
  * усиливать коммуникации с пользователями с низкой удовлетворённостью
  * использовать `DaySinceLastOrder` для выбора момента retention-кампании
  * тестировать cashback и промокоды как инструменты удержания
  * строить сегменты клиентов по вероятности оттока и потенциальной ценности

## Ограничения проекта

  * данные являются учебными и не отражают полный production-контекст
  * нет истории заказов на уровне транзакций
  * нет информации о маркетинговых коммуникациях
  * не учитывается сезонность
  * стандартный threshold 0.5 не оптимизирован под бизнес-цель
  * для production-решения нужна проверка через A/B-тест
